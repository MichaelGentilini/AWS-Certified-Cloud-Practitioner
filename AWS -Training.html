What is Cloud Computing?

the on-demand delivery of compute power, database, storage, applications, and other IT resources via the internet with pay-as-you-go pricing.

Cloud Computing Models
  • IAAS - Infrastructure as a Service
    • contains the basic building blocks for cloud IT and typically provide access to networking features, computers (virtual or on dedicated hardware), and data storage space.
    • provides you with the highest level of flexibility and management control over your IT resources and is most similar to existing IT departments
    • ðŸ’¡ EC2
  • PAAS - Platform as a Service
    • ðŸ’¡ LAMBDA
    • removes the need for organizations to manage the underlying infrastructure and allow you to focus on the deployment and management of your applications.
  • SAAS - Software as a Service
    • a completed product that is run and managed by the service provider
    • you do not have to think about how the service is maintained or how the underlying infrastructure is managed, BUT ... you have the least flexibility
    • ðŸ’¡ Gmail
Deployment Models
  • Cloud

    • - A cloud-based application is fully deployed in the cloud and all parts of the application run in the cloud.

  • Hybrid

    • - is a way to connect infrastructure and applications between cloud-based resources and existing resources
    - most common method is between the cloud and existing on-premises infrastructure to extend, and grow, an organization's infrastructure into the cloud while connecting cloud resources to internal system

  • On-Premises "Private-Cloud"

    • - Deploying resources on-premises, using virtualization and resource management tools .
    - In most cases this deployment model is the same as legacy IT infrastructure while using application management and virtualization technologies to try and increase resource utilization

S3 - Simple Storage Solution

    Amazon S3 Data Consistency Model
    • provides read-after-write consistency for PUTS of new objects in all regions
    • which means the object can be accessed immediates after copied (PUT) in an S3 Bucket
    • offers eventual consistency for overwrite PUTS and DELETES in ALL regions
    • aka.. if an S3 Bucket object is updated, it will eventually propogate and be visible eveywhere
    Amazon S3 Features - Storage Classes
    • Standard Storage Class - used for performance-sensitive and frequently accessed data (active website)
    • Standard_IA & Onezone_IA - Infrequently accessed
      • designed for long-lived and infrequently accessed data
      • plan to store data for a minimum of 30 days
    • Standard_IA is stored redundantly across multiple geographically separated AZs, thus the objects are resilient if an AZ is lost. Onezone_IA is stored in one AZ and thus is less resilient.
    • Glacier - has a minimum duration time of 90 days and can be accessed in 1-5 minutes using expedited retrieval
    • Deep_Archive has a minimum duration time of 180 days and has a default retrieval time of 12 hours. This is the lowest cost option available.
    Amazon S3 Features Additional

    • Cross-Region Replication (CRR) - enables automatic, asynchronous copying of objects across buckets in different AWS Regions.
    • - Buckets configured for cross-region replication can be owned by the same AWS account or by different accounts.

Amazon EC2 Auto Scaling

EC2 Auto Scaling helps you ensure that you have the correct number of Amazon EC2 instances available to handle the load for your application (up or down)

example
  • EC2 Instances are grouped into Auto Scaling Groups (ASG)
    • Minimum number of EC2 instances
    • Desired number of EC2 instances
    • Maximum number of EC2 instances
  • Scaling policies will automatically launch or terminate instances as demanded by the application
  • An ASG contains a collection of Amazon EC2 instances that are treated as a logical grouping for the purpose of automatica scaling and management
  • Maintaining the number of instances in an ASG and automatic scaling are the core functionalities of the Amazon EC2 Auto Scaling service
  • The size of the ASG depends on the number of instances you set as the desired capacity
  • The ASG starts by launching the desired number of EC2 instances (desired capacity; i.e. 3 EC2 instances)
  • Scaling Policies are used to increase or decrease the number of instances in a group dynamically to meet changing conditions
  • When the Scaling Policy is in effect, the ASG asjusts the desired capacity of the group betwen the minimum and naximum capacity values and launches or terminates instances as needed
  • Launch Configuration is an instance configuration template that an ASG uses to launch EC2 instances
  • Included in a Launch Configuration:
    • AMI ID
    • Instance type

    • Security group

Pricing

AWS provides agility, helps reduce IT costs and reach global coverage in minutes so you can optimize your costs continuously in order to match your needs with your environment.
AWS offers pay-as-you-go, on-demand pricing, with the best ROI for each specific use case

Link to Lab
    Pricing Key Principles
    • Understand the fundamentals of pricing
    • Start early with cost optimization
    • Maximize the power of flexibility
    • Use the right pricing model for the job
    Fundamentals of Pricing
      for the vast majority of AWS services, the following impact the cost: compute, storage, and outbound data transfer
    • Compute is paid by the minute or hour
    • Storage is paid on a per GB basis
    • NO CHARGE for data transfer INBOUND, outbound is charged and aggregated on your bill per GB (AWS Data Transfer Out)
    Start early with cost optimization
    • You should start managing your costs from the beginning of your implementations start date; the complexity grows as your move forward and scale your project
    • it's easiest and recommended to put cost visibility and control mechanisms in place before the environment becomes too large and complex
    Maximize the power of flexibility
    • With AWS you pay for exactly what you need with no minimum commitments or contracts
    • You can choose to save money through a reservation model
    • Using a pay-as-you-go model, procurement complexity is reduced which enables your business to be fully elastic
    • You don't pay for services that are not running (cost efficiency)
    Use the right pricing model for the job
    • With AWS you can coose the pricing model that best fits your business needs
    • Different pricing models are available for EC2
      • On-Demand: pay and use EC2 with no up-front payment or Long-term contract
      • Dedicated Instances: AWS hardware is dedicated to you, you don't split the host with any other customers
      • Spot Instances: purchase spare computer capacity at a discounted hourly rate
      • Reservations: pay for compute capacity ahead of time and receive up to 75% (EC2, RDS, DynamoDB, etc)

AWS WAF, Sheild and Firewall Manager

AWS WAF is a web application firewall that helps protect your web applications from common web exploits that could affect application availability, compromise security, or consume excessive resources.

Link to Lab
    AWS WAF - Web Application Firewall
  • Monitors connections forwarded to your web application
  • A WAF is a protocol 7 defense (OSI Model) and is not designed to defend against all types of attacks
  • As opposed to a typical newtwork firewall, WAFs understand traffic from the application perspective; with WAF you can monitor HTTP and HTTPS traffic, which is more than just TCP protocol and 80/443 port numbers
  • WAF typically protects web applications from attacks such as cross-site forgery, cross-site scripting (XSS), file inclusion, and SQL injection
  • This method of attack mitigation is usually part of a suite of tools which together create a holistic defense against a range of attack vectors
  • Deploying a WAF in front of a web application literally means installing a shield between the web application and the Internet users
    AWS WAF Configuration Options
  • Allow all traffic except specific requests
  • Block all traffic except requests you permit
  • Monitor and count requests with properties defined
    AWS WAF Benefits
  • Additional level of security for your web applications
  • Define custom rules to protect web apps
  • Use WAF API for automated administration
    What is AWS Shield
  • AWS Shield helps you protect against DDoS attacks
  • A denial-of-service (DoS) attack is a type of cyber attack in which a hacker aims to make a computer or other server unavailable to its users by interrupting the device's normal function
  • A distributed denial-of-service (DDoS) attack is a DoS attack that comes from many distributed sources
  • AWS Sheild helps you stay protected from DDoS attacks
    What is AWS Firewall Manager?
  • AWS Firewall Manager simplifies your AWS WAF administration and maintenance tasks across multiple accounts and resources
  • With AWS Firewall Manager, you set up your firewall rules just once
  • The service automatically applies your rules across your accounts and resources, even as you add new resources

  • Migrating applications to AWS, even without significant changes ("lift and shift"), provides organizations the benefits of a secured and cost-efficient infrastructure
  • Architectures need to be changed and get updated, which will lead to immediate benefits like agility and elasticity, that are possible and available with cloud computing
  • Following are the best practices that have emerged, as a result of cloud - traditional computing differences
      • IT Assets Become programmable Resources
    • In a traditional data center, resource provisioning is done by guessing and making assumptions on maximum peak load; this results in either idle expensive resources not being utilized or insufficient capacity to handle traffic
    • This is totally different with Cloud Computing; use the right amount of capacity, dynamically scale up or down when needed, pay-as-you-go and only pay for what you use
    • AWS services are up and running in minutes, you can use as much or little as needed
      Global, Available and Unlimited Capacity
    • When you deploy your app in the cloud, several best practices should be followed: proximity to end users, compliance or data residency constraints, costs
    • In order to achieve low latency for your application, you may want to use Amazon CloudFront CDN
    • High availability and fault tolerance for your apps using AWS global infrastructure, deploy in multiple DCs
    • With AWS there is virually unlimited capacity
      Higher Level Managed Services
    • AWS services are instantly available to use; compute, storage, databases, analytics, deployment services
    • Using managed services from AWS, helps you lower operational complexity as well as cost
    • Reducing risk for your project implementations is easy, as all AWS managed services are designed for scalability and high availability
      Security Built In
    • With AWS Cloud, governance capabilities that enable continuous monitoring and configuration changes to your IT resources are always on and available
    • This is different than traditional infrastructure, where auditing processes are a periodic and manual process
    • Solution architects can use many native AWS security and encryption features and services, which leads to meeting higher levels of compliance and data protection

Architecture - Design Principles - Scalability
    Scalability Overview
  • Systems that are expected to grow over time need to be built on top of a scalable architecture
  • Scalable architecture provides the ability to grow your environment when this is needed (additional users, traffic throughput)
  • Cloud computing allows virually unlimited growth, but the underlying architecture must be designed to support this
  • Scale either vertically or horizontally
    Scaling Vertically
  • Means increasing the capacity of your current server, as an example
  • At some point you discover that your current server can no longer process the amount of data that is constantly increasing => you need to scale/grow
  • For Example: you are running your website on an Amazon EC2 instance: a1.medium (1vCPU, 2GB RAM) and will migrate your EC2 instance to a more advanced options m5.24xlarg(96vCPU, 384GB RAM)
    Scaling Horizontally
  • Means increasing the number of resources i.e. adding more EC2 instances
  • This is not always possible, depending on the underlying architecture, which can or cannot distribute traffic to multiple resources
    Scaling Horizontally - Stateless Applications
  • Stateless vs Stateful
  • Stateless applications do NOT store any data and the connections independent from one another
  • A Stateless Application is an application that needs no knowledge of previous interactions and stores no session information; i.e. app provides the same response, to any user with the same input
  • Stateless applications are a great candidate for horizontal scaling; simply add more EC2 instances in order to run the app and terminate the instances when no longer needed
  • The easiest and most popular way to distribute traffic to a fleet of instances is using an Elastic Load Balancers ELB
    Scaling Horizontally - Stateless Components
  • Most applications need to maintain some kind of state information (web apps need to track whether a user is signed in)
  • Some web applications use HTTP cookies to store data on the client side while other scenarios require storing larger files
  • For the second option, Amazon S3 or EFS could be used
  • There are cases when you cannot change all your components in your architecture to stateless
  • Example: real-time multiplayer online gaming, users are connected to the same server (low latency and best experience this way)
  • horizontal scaling can be achieved in this case also using "session affinity" - binding all connections from a specific user to only a single server
    Scaling Horizontally - Distributed Processing
  • This is similar to breaking a problem into smaller pieces
  • When a single compute resource cannot process information, because it's too large, then the work will be distributed and split into fragments, to more instances
  • This use case is common for Big Data scenarios (processing large volume data sets)
Architecture - Design Principles - Disposable Resources instead of Fixed Servers

Disposable Resources refers to temporary resources, easily available, one time use resources

  • In traditional data center environments, you work with fixed resources or servers; this translates into high up-front costs and additional time leading to production
  • Within AWS, you launch as many servers as you need, user them as along as necessary and pay accordingly
  • Other aspects to note with regards to fixed servers
    • configuration drift and immutable infrastructure
  • Configuration Drift - configuration changes and software patches can be applied inconistently; which leads to differences in configurations (possibly untested) on your resources in the DC
  • Immutable Infrastructures - can solve the above issue; instead of patching and modifying intial configuration on your servers, just change the old server with the new one that new software packages applied
    Infrastructure Instantiating - Automation
  • Manually setting up your infrastructure is time consuming and is also error prone
  • Ideally, any new environment setup or scaling up of existing infrastructure should be done automatically
  • With AWS, you can use Bootstrapping or Golden Images, or both at the same time
    Bootstrapping

    When you launch an EC2 AMI, the instance startes with a default configuration

  • Want to configure the EC2 instance as a web server?
    • sudo su
    • service install httpd
    • server httpd start
    • cd /var/www/html
    • nano index.html
  • Run a Bootstrap script for all of the above
    Golden Image
  • Once your web server is up and running, with all security and OS patches applied, you can create a snapshot of this EC2 instance
  • The snapshot Golden Image may be used in order create another AMI
  • The AMI could be used for example in an Auto Scaling Group so that resources sustaining your app can scale up and down as needed

Architecture - Design Principles - Loose Coupling

IT systems should ideally be designed in a way that reduces inter-dependencies. Your components need to be loosely coupled to avoid changes or failure in one of the components from affecting others.

ðŸ’¡ If your Apple IPODâ€™s battery dies, you need to replace the whole device as you canâ€™t simply change the battery! That can be an example of a tightly coupled system!
    Well Defined Interfaces
  • Communication between the components should be implemented through open-source mechanisms
  • Using open-source communication interfaces (not vendor specific) leads to the possiblity of developers modifying and adapting configurations on the fly, during or after implementation
    Service Discovery
  • Implementing loose coupling means you will have a lot of services that need to either communicate with each other or other services in your environment
  • There needs to be a way to address, or call any service, in a unique way, loosely, so that no inter-dependencies are created
  • As an example, with Load Balancers, you can call a load balancer by using the endpoing name (!Not IP address)
    Asynchronous Integration
  • Asynchronous Integration refers to integrations between different services in your infrastructure
  • If 2 services can work independently of one another while together as a system then the system is considered to be asynchronous
  • As an example, service A is SNS (email) while service B could be the SQS (queuing system)
    Graceful Failure is another method to increase loose coupling
  • When a failure occurs, communication of the failure should be performed in the system and all components should be aware
  • Rerouting of traffic to healthy services should take place (Route 53 can reroute client's traffic to a healthy EC2 instance that hosts your app/website)
Architecture - Design Principles - Services, NOT Servers
You should use AWS managed services to migrate to serverless architecture as much as possible
  • Managed services, which means the services are managed by AWS making your life easier, include databases, machine learning, analytics, queueing, email, notifications and more ...
  • As an example, Amazon S3 can store literally any amount of data without worrying about capacity, availability, data replication, etc
    Serverless this means you can run your application code without a server
  • AWS Lambda is the AWS compute service that will run your code on your behalf using AWS architecture. As a bonus, it's cheaper than traditional Cloud Computing
  • With AWS Lambda, you are charged for every 100ms your code executes and the number of the times the code is triggered
